# WorkshopLM UI concept directions

Generated on July 16, 2026 after the current implementation screenshots failed the professional-quality visual bar.

## Boards

- [`workshoplm-concepts-01-05.png`](workshoplm-concepts-01-05.png): Focus Canvas, Editorial Workspace, Object Spotlight, Command Workbench, Studio Timeline
- [`workshoplm-concepts-06-10.png`](workshoplm-concepts-06-10.png): Storyboard Cinema, Created Work Gallery, Conversation to Canvas, Source-Led Studio, Unified Workshop

These are design targets, not implementation evidence. They preserve the existing product skeleton while testing materially different information architectures in the official Apps in ChatGPT visual language.

## Recommended system

Do not implement one board literally. Use one coherent system assembled from the strongest compatible directions:

- **Core Workshop:** `1. Focus Canvas` — the Map is the dominant thinking surface; Sources become a quiet horizontal shelf; conversation remains available through the ChatGPT composer.
- **Brief and Style:** `2. Editorial Workspace` — approvals and source-backed synthesis read as professional editorial work, not a settings form.
- **Created work:** `7. Created Work Gallery` — outputs are visually led, art-directed, and browsable without a dashboard grid or permanent controls.
- **Source inspection:** `9. Source-Led Studio` — citations open as a contextual bottom drawer while the current object remains dominant.
- **Storyboard:** `6. Storyboard Cinema` — the current frame is cinematic, the filmstrip is subordinate, narration is edited inline, and approval is a single clear action.

## Rejected patterns

- The current three-surface shell with persistent edge controls
- Large undirected beige canvas space
- Generic cards as the primary information architecture
- Simultaneously visible navigation, source management, workflow status, editing controls, and output controls
- Tiny artifact previews surrounded by empty chrome
- Repeated labels, badges, or helper copy that explain state instead of making state obvious

## Implementation bar

The next UI implementation is accepted only when real screenshots demonstrate:

1. One dominant current object at every step.
2. A single obvious next action.
3. Sources visible or reachable in one interaction without owning a permanent column.
4. Created work displayed at a professional review scale.
5. Storyboard frames large enough to judge and edit.
6. No screen that reads as a wireframe, admin dashboard, or component inventory.
7. Desktop and compact layouts that preserve hierarchy instead of merely fitting.
