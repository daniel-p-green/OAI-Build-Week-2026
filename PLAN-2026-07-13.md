## Local runtime

- `pnpm dev` starts the Next.js web app and local worker together.
- SQLite in WAL mode stores Workshop, version, approval, dependency, and job state.
- A configured local data root stores imported sources and generated artifacts by content hash.
- The ChatGPT/Codex in-app browser opens the localhost app and is the primary interface for implementation verification and demo capture.
- The repository is also a locally installable unified plugin with a WorkshopLM skill, bundled stdio MCP server, and compact status/trace widgets.
- The plugin requires no connected app for the judge fixture. Granola/Drive integrations remain optional adapters.
- `pnpm demo:reset` recreates the sanitized judge fixture without relying on existing machine state.
- `pnpm demo:e2e` defaults to recorded-fixture mode so the demonstrated seam can be replayed and diagnosed without paid calls; live-provider checks remain separate. The fixture supports reliable video capture and optional inspection, not a parallel judge-installation product.
- No hosting, separate WorkshopLM login system, cloud database/storage, remote worker, or HyperFrames service is required. ChatGPT/Codex account state comes from the local Codex app server.

## Integration spikes

Run these in isolated `spikes/` workspaces before product code depends on provider or host-surface assumptions. Spike code is disposable evidence; retain fixtures, measured results, and adapter lessons. Each spike must end with a dated `log.md` entry stating pass, fallback, or credential blocker.

### Spike A — ChatGPT account, task, and voice capture

**Location:** `spikes/a-host-sync/`

**Proves:** Codex app-server `account/read` and supported ChatGPT login flow, linkage between the current ChatGPT task and a Workshop, durable ingestion of native typed/voice turns, and whether a custom Realtime fallback is necessary.

**Verification:** `pnpm spike:host-sync` connects server-side to the local app-server, reads account state without exposing tokens, links a disposable task to a Workshop, persists typed and native voice-originated turns when the host exposes them, and writes a sanitized capability report.

**Pass criteria:**

1. `account/read` returns an authenticated ChatGPT account or the supported login URL flow.
2. Browser code receives only safe account display state, never access/refresh tokens.
3. One typed turn and one native voice-originated turn link to the correct Workshop with stable task/turn provenance when supported.
4. Reconnect/restart does not duplicate acknowledged turns.
5. The report makes an explicit native-path versus Realtime-fallback decision.

**Fallback:** if native voice-turn synchronization is not proven by July 14 end of day CT, automatically add a narrow in-browser `gpt-realtime-2.1` WebRTC capture control with server-minted ephemeral secret and the same transcript contract. Place it in the host strip or top bar and label it as capture-only; it is not a text composer or second conversation UI. ChatGPT remains the primary conversation surface.

**Decision, July 15:** the deadline elapsed without proof of native durable task/voice synchronization. The capture-only `gpt-realtime-2.1` fallback is final for the demo. Its durable transcript/source contract and bounded UI are implemented; live WebRTC transport remains the open implementation and verification item.

### Spike B — local plugin grounding

**Location:** `spikes/b-filesearch/`

**Proves:** local normalization with durable locators, standard MCP `search`/`fetch` tools, FTS5/BM25 plus exact retrieval, claim-level citations, and consistent evidence access from ChatGPT and the production worker.

**Verification:** `pnpm spike:grounding` parses three sanitized fixtures, builds the local index, runs exact and paraphrased queries through MCP `search`/`fetch`, then asks GPT-5.6 for a grounded response from the returned evidence bundle.

**Pass criteria:**

1. All fixture files normalize into deterministic chunks and locators.
2. Exact questions and hostile paraphrases with zero intentional lexical overlap return inspectable ranked results.
3. At least three response citations resolve bidirectionally through claim → chunk → source.
4. ChatGPT/Codex and the worker receive the same result contract.
5. Unsupported or absent evidence produces `unverified`, never a fabricated citation.
6. GPT query-expansion latency, token use, and estimated cost per retrieval are recorded because expansion sits on the grounded-turn hot path.

**Fallback/optional comparison:** if FTS5 plus GPT query expansion misses the paraphrase fixture, add locally stored semantic vectors with bounded in-process cosine search. Hosted OpenAI `file_search` may be compared behind the same adapter but is not required.

### Spike C — GPT Image 2 coherent batch

**Location:** `spikes/c-images/`

**Proves:** six-image generation, shared Visual DNA and locked-reference inputs, per-image provenance, selective regeneration, persistence, and real latency/cost envelope.

**Verification:** `pnpm spike:images` generates six assets and then regenerates only panel four; the report compares asset IDs and metadata before and after.

**Pass criteria:**

1. Six images persist with model, prompt, reference, timing, and version metadata.
2. Regeneration replaces only the selected image.
3. Shared Visual DNA and reference inputs fit provider limits.
4. Actual latency and cost are recorded. A median above 90 seconds changes demo queue behavior and fixture strategy; it does not by itself fail the capability.

**Fallback:** reduce the live judge batch to four and generate sequentially while retaining a pre-generated six-image sanitized fixture. If model access fails, use the best available image model behind the same adapter and soften judge-facing claims.

### Spike D — local HyperFrames render

**Location:** `spikes/d-video/`

**Proves:** an approved three-panel storyboard becomes a deterministic narrated MP4 through the local HyperFrames CLI, with synchronized captions, transitions, brand tokens, and visible AI-voice disclosure.

**Prerequisites:** Node.js 22+, FFmpeg, and a minimal committed `spikes/d-video/DESIGN.md`. HyperFrames runs entirely locally as an HTML/Chrome/FFmpeg workflow. Do not implement HyperFrames API/auth, remote jobs, polling, or downloads.

**Verification:**

1. `npx hyperframes doctor`
2. `npx hyperframes lint`
3. `npx hyperframes validate`
4. `npx hyperframes inspect`
5. `npx hyperframes render --output artifacts/spike-d.mp4`
6. `ffprobe` verifies duration, video stream, and audio stream.

Use `gpt-4o-mini-tts` panel narration when credentials are available; otherwise use a deterministic local audio fixture to prove composition, then run the live TTS seam separately.

**Pass criteria:**

1. A playable local MP4 exists with video and audio streams.
2. scene timing, captions, and transitions match the fixture;
3. the AI-voice disclosure is visible;
4. render time and output metadata are recorded.

**Fallback:** a minimal FFmpeg composition of approved panel images, captions, and narration behind the same `renderVideo` adapter. Do not introduce Remotion unless both HyperFrames and the FFmpeg fallback fail for a reason Remotion specifically resolves.

### Spike E — unified plugin host surfaces

**Location:** `spikes/e-plugin/`

**Proves:** the current `.codex-plugin/plugin.json` package loads locally, its WorkshopLM skill is discoverable, its bundled stdio MCP server starts without authentication, and its tools/widgets are available on the ChatGPT Work and Codex surfaces actually installed for the demo.

**Verification:** install/load the local spike plugin through the supported desktop development path, invoke its skill from Work and Codex, call a read tool and an ineligible write tool, and capture the visible result plus sanitized MCP logs.

**Pass criteria:**

1. The plugin appears with correct name, description, prompts, and assets.
2. The skill activates from both available surfaces.
3. Codex can call the bundled local stdio tool and render its compact widget.
4. Actual Work support for local stdio tools is recorded rather than assumed.
5. No app connection, OAuth flow, or remote MCP endpoint is required for the verified core path.

**Fallback:** if Work cannot invoke a bundled local stdio server, keep the unified skill available in Work but route the hackathon's executable tool path through Codex and the local in-app browser. The full product and judge path remain local; do not add hosting merely to force surface parity.
