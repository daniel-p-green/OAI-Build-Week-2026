# Proposed six-phase closeout sequence

**Status:** Rough planning reference; not yet ratified and does not replace `GOAL.md` or `PLAN-2026-07-13.md`.

**Captured:** July 15, 2026

**Source:** Founder-supplied planning note.

Each phase has a goal, a checklist, and an exit condition. A phase does not open until the previous phase's exit condition is met because each phase depends on artifacts from the one before it.

## Phase 1 — Make it real (founder actions)

**Goal:** Every remaining unknown is resolved and the product runs on live AI.

- [ ] Close Spike A: log entry declaring the capture-only fallback the final voice path; check the box.
- [ ] Authorize provider spend with a ceiling.
- [ ] Run the nine-request GPT-5.6 benchmark; let its artifact set the Sol/Terra/Luna operation defaults.
- [ ] Run `demo:live --execute`; inspect every image, narration clip, provenance record, and the narrated MP4 before any checkbox moves.
- [ ] Run the Spike E fresh-task invocation from the desktop host; log the verified surface (it controls all README/Devpost/video platform claims).
- [ ] Capture the primary Codex `/feedback` Session ID from the most representative session; log which and why. If the surface cannot produce one, contact organizers immediately.
- [ ] Record the raw founder brainstorm, honestly dated and labeled.

**Exit:** Zero open items require founder-only access; live provider assets exist and are inspected.

## Phase 2 — Integrate the real assets

**Goal:** The live assets flow through the product, and verification reflects them.

- [ ] Wire real image bytes, narration, and the narrated render through the gallery, Storyboard, and Outputs.
- [ ] One-time visual-baseline refresh, logged as expected content change, not regression.
- [ ] Red-team pass on the live-provider path: gate bypass, stale narration, partial image failure, retry.
- [ ] Full acceptance run green with provider-backed content.
- [ ] Check the GPT-5.6, image batch, and narration boxes in `GOAL.md` only against inspected artifacts.

**Exit:** Acceptance run green end to end with live content; no `planned` labels remain except where honestly true.

## Phase 3 — Recording loop (repeat until the film is right)

**Goal:** A demo film that proves all four judging criteria in under three minutes.

Each cycle:

- [ ] Fixture reset, one rehearsal, full recording from the intended in-app surface.
- [ ] Watch it as a judge; write the critique list (stumbles, labels needing narration, dead seconds).
- [ ] Fleet burns down the critique list; acceptance stays green at every merge.
- [ ] Re-record.

Film requirements:

- [ ] Live GPT-5.6 shown on screen, not just claimed.
- [ ] Both approval gates, one edit demonstrating control, and one plugin-widget doorway moment.
- [ ] Evidence trace pulled from an output block to a source sentence.
- [ ] Brainstorm-to-finished-submission provenance reveal.
- [ ] Codex/GPT-5.6 disclosure matching evidence exactly; AI-voice disclosure visible.
- [ ] Target 2:40–2:45, with a 2:55 ceiling; no Google marks in title, thumbnail, or UI; clear audio; no dead time.

While recording, also capture portfolio ephemera: session views, intermediate states, and director's-cut footage.

**Exit:** A take that can be shown to a judge without qualification.

## Phase 4 — Meta-demo closes its own loop

**Goal:** WorkshopLM produces its own submission package, traced.

- [ ] Run the brainstorm plus project sources through the live product; produce one traced Output set: Devpost description, README narrative, deck, thumbnails, Storyboard, and video assets.
- [ ] Finalize the provenance surface: log timeline, self-correction event, commits, Session ID, and time to first output.
- [ ] Finalize README: install, fixture commands, supported platform per Spike E, sample data, and truthful Codex/GPT-5.6 disclosure.
- [ ] Tag the judge release commit; point every instruction at the tag, not `main`.

**Exit:** The submission package exists as WorkshopLM outputs with inspectable trace.

## Phase 5 — Freeze, stage, verify

**Goal:** Nothing changes anymore; everything is checked cold.

- [ ] Feature freeze; polish and reliability only, on regression evidence.
- [ ] Final cut edited and uploaded to YouTube, set public; verify logged out.
- [ ] Create the Devpost project; fill every field: name, tagline, description, category (Work & Productivity), submitter type, country, repo URL, judge instructions, Session ID, plugin install and test path, and video URL. Leave it in draft.
- [ ] Cold verification: every link in a private session, fresh in-app-browser run of the fixture, README read as a stranger, tagged release installs and runs clean, and repo permissions confirmed.
- [ ] Re-check the official rules and latest announcements one final time.

**Exit:** Every field is filled and every link is verified by someone acting as if they have never seen the project.

## Phase 6 — Submit, then convert to portfolio

- [ ] Submit; confirm status reads `Submitted`, not `Draft`.
- [ ] Keep the tagged release, fixture, video, and repo access stable through the winner announcement.
- [ ] Write the case study while fresh: the bet, the operating system, the self-correction, real numbers, and honest limits.
- [ ] Cut the practitioner director's cut from the recording-session footage.
- [ ] Add a `Start here` section to the repo pointing cold visitors at `GOAL.md`, the log, the self-correction entry, and the case study.

**Exit:** The submission is live, and the project is legible as a standalone case study to someone who never heard of the hackathon.

## Standing rules

- The acceptance run gates every merge.
- No checkbox moves without evidence that proves the exact claim.
